# 数据库连接池耗尽处理 Runbook

## 适用场景与检索关键词

- 告警：`DatabaseConnectionPoolExhausted`、等待连接超时、活跃连接达到上限。
- 关键词：connection pool exhausted、timeout waiting for connection、Too many connections、connection leak、pool wait、active connections、pending threads。
- 适用于应用获取数据库连接阶段变慢或失败；SQL 已获得连接但执行慢应使用数据库慢查询 Runbook。

## 告警定义与影响

连接池活跃连接长期接近最大值、等待队列持续增长或获取连接超时均应告警。连接池耗尽会让请求在执行 SQL 之前排队，最终产生超时和 5xx。盲目增大应用连接池可能超过数据库最大连接数并扩大故障。

## 排查前必须确认的信息

1. 服务、实例数、每实例连接池上下限和数据库最大连接数。
2. 获取连接等待时间、SQL 执行时间、活跃/空闲连接和等待线程数。
3. 是否刚扩容应用、发布新版本或调整连接池配置。
4. 是否存在慢事务、连接泄漏、数据库重启或网络中断。
5. 所有应用实例理论最大连接数之和是否超过数据库预算。

## 证据驱动的排查流程

### 1. 确认关联告警和资源

使用 `get_current_time`、`query_prometheus_alerts` 获取活动告警；使用 `query_cpu_metrics`、`query_memory_metrics` 检查应用是否因等待线程堆积消耗资源。

### 2. 查询连接池日志

通过 `search_topic_by_service_name`、`get_topic_info_by_name` 找到日志主题，再用 `search_log` 检索 pool exhausted、timeout waiting、Too many connections、connection leak、borrow/return、transaction timeout 和 database unavailable。

### 3. 建立连接容量账本

计算实例数乘以每实例最大连接数，加上其他服务和运维连接，和数据库允许连接数比较。应用自动扩容可能让总连接预算瞬间超限。

### 4. 人工确认数据库会话

由 DBA 查看活动会话、空闲事务、长事务、连接来源和等待事件。不得由 Agent 自动终止数据库连接。

## 现象、证据与结论判断表

| 现象 | 必要证据 | 可支持的判断 |
|---|---|---|
| pool active 达上限且 SQL 耗时升高 | 连接池指标、SQL 耗时 | 慢查询占用连接过久 |
| 活跃连接增长后不归还 | borrow/return、泄漏检测、代码路径 | 连接泄漏 |
| 应用扩容后立即出现 Too many connections | 实例数、池上限、数据库上限 | 总连接预算超限 |
| 数据库重启后大量连接同时重建 | 数据库事件、重连日志 | 重连风暴 |
| 获取连接慢但数据库会话很少 | 应用池状态、网络错误 | 应用池锁、驱动或网络问题 |
| 长事务保持 idle in transaction | 会话状态、事务时长 | 事务未及时提交或回滚 |

## 常见原因与处理方案

### 慢 SQL 或长事务占用连接

识别持有连接时间最长的 SQL 和事务，转入慢查询排查。设置事务超时并缩小事务边界，不要只增大连接池。

### 连接泄漏

启用连接泄漏检测，核对所有异常分支是否归还连接。短期可滚动重启释放连接，长期必须修复生命周期。

### 总连接预算错误

根据数据库容量重新分配各服务连接池，并把应用扩容上限纳入预算。必要时先降低单实例上限，再有边界地扩容。

### 数据库或网络抖动引发重连风暴

设置带抖动的指数退避和连接建立速率限制，避免所有实例同时重连。

## 分级处置与风险控制

- 等待连接偶发且业务正常：观察峰值和连接归还。
- 等待队列增长：限流、暂停批处理、处理慢事务或泄漏。
- 修改连接池上限必须同时验证数据库连接预算，分批发布并可回滚。
- 终止会话、修改数据库最大连接数或重启数据库属于高风险 DBA 操作。
- 核心请求大量超时或数据库连接接近硬上限时，立即升级人工 DBA 和应用负责人。

## 修复验证

1. 获取连接等待时间和等待线程数恢复到基线。
2. active、idle、pending 连接比例稳定，连接能够正常归还。
3. 数据库总连接数保留安全余量，不再出现 Too many connections。
4. 接口延迟和 5xx 恢复，慢事务和泄漏告警消失。
5. 应用扩缩容后连接预算仍满足约束。

## 常见误判

- 不能仅凭数据库连接错误认定数据库宕机，可能是应用连接池自身耗尽。
- SQL 执行慢和获取连接慢是两个阶段，必须分别计时。
- 增大连接池可能短暂降低等待，但会增加数据库压力。
- 重启应用会清空连接池，却不能修复连接泄漏或慢事务根因。

## 预防措施

- 监控 active、idle、pending、acquire time 和 timeout。
- 建立数据库总连接预算并关联应用最大实例数。
- 启用泄漏检测、事务超时和连接最大生命周期。
- 重连采用指数退避、抖动和速率限制。
