# HTTP 5xx 错误率过高告警处理 Runbook

## 适用场景与检索关键词

- 告警：`HighErrorRate`、HTTP 5xx 比例升高、请求成功率下降。
- 关键词：HTTP 500、502、503、504、Internal Server Error、Bad Gateway、Service Unavailable、Gateway Timeout、exception spike。
- 适用于服务仍有请求进入但错误比例异常；全部实例无法访问应同时参考服务不可用 Runbook。

## 告警定义与影响

建议按服务 SLO 定义，例如 5xx 比例连续 5 分钟超过 5%，或成功率低于 99%。错误可能由应用异常、网关无健康后端、依赖超时、容量不足或发布回归引起。不同状态码指向不同层次，不能把所有 5xx 都解释为应用代码错误。

## 排查前必须确认的信息

1. 服务、接口、环境、状态码分布和告警时间窗口。
2. 错误影响所有实例还是集中在某个版本、实例或地域。
3. 同期 QPS、响应时间、CPU、内存和最近发布记录。
4. 失败请求的 request ID、trace ID、客户端类型和入口网关。
5. 错误是持续增长、突发尖峰还是固定周期出现。

## 证据驱动的排查流程

### 1. 确认活动告警

使用 `get_current_time`、`query_prometheus_alerts` 确认告警状态和影响范围。当前无活动告警时，只能描述为历史或用户侧现象。

### 2. 查询资源趋势

使用 `query_cpu_metrics`、`query_memory_metrics` 检查错误上涨前是否出现资源饱和、OOM 或指标归零。资源正常不代表应用逻辑正常，资源高也不自动成为根因。

### 3. 查询错误日志

通过 `search_topic_by_service_name` 和 `get_topic_info_by_name` 定位日志主题，再使用 `search_log` 查询故障窗口。按状态码、exception、timeout、connection refused、pool exhausted、upstream 和 request ID 聚合，比较错误最早出现时间与发布、依赖告警的关系。

### 4. 区分入口和应用层

人工比较网关请求与实例直连结果，并检查响应头、upstream 地址和错误主体。502/504 常与网关到上游链路有关，500 更常见于应用异常，但必须以日志为准。

## 现象、证据与结论判断表

| 现象 | 必要证据 | 可支持的判断 |
|---|---|---|
| 500 集中于单接口和同一异常栈 | 日志堆栈、接口、版本 | 应用逻辑或数据问题 |
| 502 且实例直连正常 | 网关 upstream 日志、直连探测 | 网关或服务发现异常 |
| 503 且健康后端数量为零 | 健康检查、实例状态 | 实例不可用或探针错误 |
| 504 与下游耗时同步 | trace 分段、下游 timeout | 下游或网络超时 |
| 错误只在新版本实例出现 | 版本标签、发布时间 | 发布回归 |
| 错误率随 QPS 和队列上涨 | QPS、资源、队列证据 | 容量不足或过载保护 |

## 常见原因与处理方案

### 发布回归或应用异常

保留首个异常栈和失败样本，对比版本差异。证据明确时暂停发布并评估回滚，不要在无版本相关性时盲目回滚。

### 下游超时或连接失败

确认失败依赖、超时阶段和重试次数，启用有限重试、熔断或降级。无限重试会放大错误率和延迟。

### 网关或健康检查异常

核对后端池、探针路径、端口和服务发现。实例直连正常时优先修复入口层，不应重启全部应用实例。

### 资源和并发耗尽

按 CPU、内存、线程池或连接池证据进入对应 Runbook。扩容前必须评估下游容量。

## 分级处置与风险控制

- 错误低于 SLO 且短时恢复：保留样本并观察。
- 核心接口错误持续：暂停发布，限流或降级非核心功能。
- 需要回滚或重启时必须确认版本相关性和剩余容量；调用 `restart_service` 必须人工审批。
- 核心业务错误超过 20%、错误持续扩大或涉及数据错误时，立即升级人工负责人。

## 修复验证

1. 5xx 比例和成功率连续 30 分钟恢复到基线。
2. 各状态码来源明确，异常栈和 timeout 日志不再新增。
3. 核心接口通过真实业务验证，不能只检查健康接口。
4. CPU、内存、线程池、连接池和下游延迟保持正常。
5. 若执行回滚，确认旧版本没有兼容性或数据迁移问题。

## 常见误判

- 不能仅凭 HTTP 500 认定是代码缺陷，配置、数据和依赖异常也可触发 500。
- 502/504 不一定是网关自身故障，可能是上游拒绝连接或超时。
- 错误率下降可能是流量已经降为零，必须同时检查请求量。
- 单一日志样本不能代表主要错误类型，应按时间和异常指纹聚合。

## 预防措施

- 按状态码、接口、实例、版本和地域监控错误率。
- 日志保留 request ID、trace ID、依赖名称和异常指纹。
- 发布设置错误率自动暂停和回滚门槛。
- 为依赖调用设置超时、有限重试、熔断和降级。
