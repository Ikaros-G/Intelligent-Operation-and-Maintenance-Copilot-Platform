# 服务响应时间过长告警处理 Runbook

## 适用场景与检索关键词

- 告警：`SlowResponse`、`HighLatency`、`ServiceTimeout`。
- 关键词：P95、P99、response time、upstream latency、timeout、请求堆积、慢接口、长尾延迟。
- 适用于服务仍可用但响应显著变慢；大量 HTTP 5xx 应同时参考高错误率 Runbook。

## 告警定义与影响

可按业务 SLO 设置阈值，例如 P99 连续 5 分钟超过 3 秒或超时率持续升高。慢响应可能来自 CPU、内存、数据库、Redis、线程池、外部 API 或网络，端到端延迟只是结果，必须分段定位。

## 排查前必须确认的信息

1. 服务、接口、环境、时间范围、正常基线和当前 P50/P95/P99。
2. 是所有接口、单接口、单实例还是单地域变慢。
3. QPS、并发数、HTTP 错误率和客户端超时设置。
4. 是否有发布、配置变更、缓存失效、批处理或下游故障。
5. 延迟发生在入口排队、应用执行、数据库、缓存还是外部调用阶段。

## 证据驱动的排查流程

### 1. 确认告警与时间窗口

使用 `get_current_time` 和 `query_prometheus_alerts` 确认活动告警。若用户描述的是历史问题，应使用其提供的准确时间窗口，不要把当前无告警解释为问题从未发生。

### 2. 查询 CPU 和内存趋势

使用 `query_cpu_metrics`、`query_memory_metrics` 比较故障窗口和正常窗口。资源高只能说明相关性，不能单独证明根因。

### 3. 查询接口和下游日志

使用 `search_topic_by_service_name`、`get_topic_info_by_name` 定位日志主题，再用 `search_log` 检索 response_time、timeout、upstream、slow query、pool exhausted、RejectedExecutionException、retry 和 cache miss。按 request ID 或 trace ID 串联同一次请求的各阶段耗时。

### 4. 建立分段耗时

从日志或链路追踪中提取入口等待、应用处理、数据库、Redis、外部 API 和序列化耗时。没有分段数据时，应在报告中声明证据不足，并建议补充埋点。

### 5. 人工补充现场检查

```bash
curl -sS -o /dev/null -w 'dns=%{time_namelookup} connect=%{time_connect} start=%{time_starttransfer} total=%{time_total}\n' https://<endpoint>
ss -s
vmstat 1 5
iostat -xz 1 5
```

## 现象、证据与结论判断表

| 现象 | 必要证据 | 可支持的判断 |
|---|---|---|
| P99 升高但 P50 正常 | 慢请求样本、实例分布 | 长尾、单实例或少量慢依赖 |
| 所有分位数随 QPS 上升 | QPS、资源和队列趋势 | 容量不足或排队 |
| 数据库阶段耗时占比最高 | SQL 指纹、慢查询和执行时间 | 转入数据库慢查询排查 |
| 获取连接耗时高 | 连接池等待、活跃连接和超时日志 | 转入连接池耗尽排查 |
| CPU/内存正常但外部调用慢 | upstream 分段耗时、下游错误 | 下游或网络延迟 |
| 仅单实例明显变慢 | 实例维度延迟、资源和日志 | 实例局部故障或负载不均 |

## 常见原因与处理方案

### 容量不足或请求排队

确认 QPS、并发和队列长度同步上涨后，评估限流、水平扩容或异步化。扩容前检查数据库和 Redis 的连接容量。

### 数据库或连接池问题

SQL 执行慢时进入数据库慢查询 Runbook；获取连接慢、连接池等待超时时进入连接池耗尽 Runbook。增大连接池不是默认答案，可能压垮数据库。

### 缓存失效或 Redis 延迟

检查 cache miss、热点 key、大 key 和 Redis timeout。缓存恢复前可对非关键请求降级，但要防止缓存击穿把流量全部转给数据库。

### 外部依赖和网络

设置合理连接和读取超时、有限重试、熔断与隔离舱。网络问题需区分 DNS、建连、TLS 和首字节耗时。

### 线程池或锁竞争

结合队列满、拒绝执行、线程栈和锁等待证据处理。仅增加线程数可能增加上下文切换和下游压力。

## 分级处置与风险控制

- **观察级**：短暂波动且 SLO 未受影响，保留慢请求样本并观察。
- **缓解级**：限流、降级非核心功能、暂停批处理或有边界地扩容。
- **高风险级**：重启、回滚、连接池变更和线程池变更必须人工审批并具备回滚方案；若需要重启可申请 `restart_service`。
- P99 持续恶化、超时率上升、线程或连接池耗尽时，应升级人工值班处理。

## 修复验证

1. P50/P95/P99 连续 30 分钟恢复到正常基线。
2. 超时率和 HTTP 5xx 回落，请求队列不再增长。
3. 数据库、Redis、外部 API 和应用各阶段耗时均稳定。
4. CPU、内存、线程池和连接池有足够余量。
5. 使用真实核心业务请求验证，而不只检查健康接口。

## 常见误判

- 不能仅凭“接口慢”就判断 CPU 高或数据库慢，必须比较分段耗时和时间相关性。
- 平均响应时间正常不能排除 P99 长尾问题。
- 增加超时时间只会让请求等待更久，不等于性能恢复。
- 扩容应用可能增加下游连接数并使数据库或 Redis 更慢。
- 重启后短时变快可能只是清空队列或缓存，仍需验证根因。

## 预防措施

- 同时监控各延迟分位数、QPS、错误率、并发和队列长度。
- 为数据库、Redis 和外部调用记录独立耗时及 trace ID。
- 设置超时预算、有限重试、熔断和隔离策略。
- 定期压测并记录单实例安全容量和依赖容量上限。
