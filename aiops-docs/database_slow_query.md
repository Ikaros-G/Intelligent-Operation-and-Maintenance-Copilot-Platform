# 数据库慢查询处理 Runbook

## 适用场景与检索关键词

- 告警：`DatabaseSlowQuery`、SQL 执行时间超过阈值、数据库响应变慢。
- 关键词：slow query、query timeout、lock wait、full table scan、filesort、EXPLAIN、慢 SQL、索引失效。
- 适用于 SQL 已开始执行但耗时过长；等待获取连接应使用数据库连接池耗尽 Runbook。

## 告警定义与影响

可按业务设置 SQL 耗时阈值，例如超过 1 秒记录、超过 3 秒告警。慢查询会占用数据库 CPU、I/O、锁和连接，导致请求排队并形成级联超时。必须定位 SQL 指纹和执行阶段，不能仅凭接口响应慢判断数据库存在慢查询。

## 排查前必须确认的信息

1. 数据库类型、实例、库名、服务名和故障时间窗口。
2. SQL 指纹、平均/最大耗时、调用次数和影响接口。
3. 是单条 SQL、某类查询还是整个数据库同时变慢。
4. 是否有数据量增长、表结构变更、索引变更、发布或批处理。
5. 连接等待时间与 SQL 执行时间是否已分开记录。

## 证据驱动的排查流程

### 1. 确认告警和应用影响

使用 `get_current_time`、`query_prometheus_alerts` 获取活动告警。再使用 `query_cpu_metrics`、`query_memory_metrics` 查看应用是否因数据库等待出现请求堆积，而不是把应用 CPU 高直接归因于数据库。

### 2. 查询应用和慢查询日志

使用 `search_topic_by_service_name`、`get_topic_info_by_name` 定位日志主题，再用 `search_log` 查询 slow query、query timeout、lock wait、SQL 指纹、rows examined 和 affected rows。优先保留脱敏后的 SQL 模板，不在报告中暴露密码或个人数据。

### 3. 对比正常窗口

比较相同 SQL 在正常时段和故障时段的耗时、调用量与扫描行数。调用量突增和单次执行变慢是两种不同问题。

### 4. 人工数据库检查

由 DBA 在只读会话中执行对应数据库的活动会话、锁等待和执行计划检查，例如对脱敏 SQL 执行 EXPLAIN。生产环境执行 EXPLAIN ANALYZE 可能实际运行 SQL，必须先评估风险。

## 现象、证据与结论判断表

| 现象 | 必要证据 | 可支持的判断 |
|---|---|---|
| 扫描行数远大于返回行数 | 执行计划、rows examined | 缺少索引或索引未命中 |
| 同一 SQL 突然变慢且统计信息陈旧 | 计划变化、统计信息时间 | 执行计划退化 |
| 大量会话等待同一锁 | 锁等待链、长事务 | 锁竞争或未提交事务 |
| 单次耗时正常但调用量暴涨 | SQL 指纹调用次数、QPS | N+1 查询或流量增长 |
| 获取连接耗时高而执行耗时正常 | pool wait、SQL duration | 连接池问题，不是慢查询 |
| 数据量增长后排序和临时表增多 | 数据量、执行计划、磁盘临时表 | 查询或索引不适配增长 |

## 常见原因与处理方案

### 缺少索引或索引失效

根据执行计划和查询条件设计最小必要索引，评估写入放大、存储和重复索引。生产建索引前需 DBA 审批和回滚方案。

### 锁竞争和长事务

定位阻塞源、事务开始时间和业务操作。终止会话可能引起回滚压力，只能由 DBA 评估后执行。

### N+1 查询或调用量突增

通过 trace 和 SQL 指纹统计确认，改为批量查询、预加载或缓存。不要靠扩大连接池掩盖查询次数问题。

### 执行计划退化或数据倾斜

比较计划、统计信息和参数分布，评估更新统计信息、改写 SQL 或固定计划；变更前需在相近数据量环境验证。

## 分级处置与风险控制

- 低影响：记录 SQL 指纹并进入优化排期。
- 中影响：限流高成本接口、暂停批处理、使用只读副本分担查询。
- 高风险：建删索引、终止事务、修改 SQL hint 和数据库参数必须由 DBA 审批。
- 核心业务超时、锁等待扩散或数据库 CPU/I/O 接近饱和时，立即升级人工 DBA。

## 修复验证

1. 目标 SQL 的 P95/P99、扫描行数和调用量恢复到基线。
2. 锁等待、临时表和慢查询数量下降。
3. 应用接口延迟、连接池等待和错误率同步恢复。
4. 新索引未导致写入延迟或存储异常。
5. 使用代表性参数验证，防止只优化单个样本。

## 常见误判

- 不能仅凭接口慢判断 SQL 慢，可能耗时在连接池、网络或外部 API。
- 慢查询日志中的一条 SQL 不一定是主要贡献者，需要结合调用次数。
- 增加索引不是无成本操作，也可能因选择性低而不生效。
- 数据库 CPU 高可能来自批处理、复制或备份，不一定来自目标服务。

## 预防措施

- 记录 SQL 指纹、执行耗时、扫描行数和调用来源。
- 上线前在接近生产数据量的环境检查执行计划。
- 为事务设置合理超时，监控长事务和锁等待。
- 定期审查高总耗时 SQL，而不只关注单次最慢 SQL。
