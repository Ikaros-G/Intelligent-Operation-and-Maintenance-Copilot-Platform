# 服务不可用告警处理 Runbook

## 适用场景与检索关键词

- 告警：`ServiceUnavailable`、`HealthCheckFailed`、`AllInstancesDown`。
- 关键词：HTTP 502、HTTP 503、connection refused、health check failed、启动失败、端口未监听、服务宕机。
- 适用于服务完全无法访问、健康检查连续失败或所有实例下线；单纯延迟升高应先使用慢响应 Runbook。

## 告警定义与影响

健康检查连续失败、可用实例数为零或核心接口成功率接近零均属于严重可用性事件。影响可能是业务完全中断，也可能仅限单实例或单地域。必须先确认真实探测结果，不能把一次客户端超时直接定义为服务整体不可用。

## 排查前必须确认的信息

1. 服务名、环境、地域、入口域名、失败接口和故障开始时间。
2. 是一个实例、部分实例、全部实例还是仅某类客户端失败。
3. HTTP 状态码、健康检查结果、最近发布或配置变更。
4. CPU、内存、重启次数以及数据库、Redis、DNS 等依赖状态。
5. 是否已有流量摘除、自动扩缩容、熔断或回滚动作。

## 证据驱动的排查流程

### 1. 确认活动告警和影响范围

使用 `get_current_time` 记录诊断时间，使用 `query_prometheus_alerts` 获取活动告警、状态和影响实例。若返回零条告警，应报告“当前监控未发现活动告警”，并继续核实用户提供的失败请求，不能虚构服务正在宕机。

### 2. 查询资源指标

使用 `query_cpu_metrics` 和 `query_memory_metrics` 检查实例是否资源耗尽、OOM 前兆或指标突然归零。指标归零既可能是服务停止，也可能是监控采集异常，需要日志和探测共同确认。

### 3. 查询启动和请求日志

先使用 `search_topic_by_service_name` 找到日志主题，必要时通过 `get_topic_info_by_name` 确认主题，再用 `search_log` 查询故障前后 15 至 30 分钟。重点检索 startup failed、bind failed、connection refused、OutOfMemoryError、configuration error、dependency timeout、HTTP 5xx 和 shutdown。

### 4. 人工确认网络和进程状态

```bash
curl -v --max-time 5 http://<host>:<port>/health
ss -lntp
ps -ef | grep <service>
docker ps --filter name=<service>
docker inspect <container>
```

这些结果用于区分入口、网络、进程和应用层问题，不应在未确认目标环境时直接执行重启。

## 现象、证据与结论判断表

| 现象 | 必要证据 | 可支持的判断 |
|---|---|---|
| 所有实例健康检查失败且端口未监听 | 探测、进程状态、启动日志 | 应用未启动或已崩溃 |
| 仅新版本实例失败 | 发布时间、版本标签、启动异常 | 发布或配置回归 |
| 应用进程存活但请求均超时 | 线程池、下游调用、资源指标 | 线程耗尽或依赖阻塞 |
| 网关返回 502/503，但实例直连正常 | 网关日志、直连探测 | 负载均衡或服务发现异常 |
| OOMKilled 后循环重启 | 重启记录、内存趋势、exit code 137 | 容器内存问题 |
| 只有单个客户端失败 | 多点探测、DNS 和客户端网络 | 客户端或局部网络问题 |

## 常见原因与处理方案

### 发布或配置错误

对比故障时间和发布时间，检查环境变量、配置语法、端口和依赖地址。证据明确指向新版本时优先滚动回滚，保留失败实例日志用于复盘。

### 资源耗尽或进程崩溃

根据 OOM、CPU、文件描述符和磁盘证据处理对应资源问题。不要仅靠重启掩盖持续泄漏、日志盘写满或线程耗尽。

### 下游依赖不可用

识别数据库、Redis、外部 API 或 DNS 的失败模式。对非核心依赖启用熔断和降级，对核心依赖升级相应值班人员，避免无限重试放大故障。

### 网关、服务发现或网络异常

比较入口探测和实例直连结果，检查后端池、健康检查路径、DNS 和端口监听。只有入口失败时，不应修改应用本身。

## 分级处置与风险控制

- **单实例故障**：先摘除故障实例并保留证据，确认剩余容量。
- **部分实例故障**：暂停发布，评估回滚，避免健康实例被过载。
- **全部实例故障**：启动严重事件响应，优先恢复最小可用能力并同步影响范围。
- 使用 `restart_service` 前必须明确目标服务、原因、剩余容量和回滚路径，并经过人工审批。
- 5 分钟内无法确认影响范围、15 分钟内无法恢复核心功能或涉及数据一致性时，立即升级人工负责人。

## 修复验证

1. 所有目标实例健康检查连续通过，入口和实例直连均正常。
2. 核心业务请求成功率恢复，HTTP 5xx 降到基线。
3. CPU、内存、线程、连接池和重启次数稳定。
4. 启动失败、依赖连接错误和异常退出日志不再新增。
5. 至少观察 30 分钟，并验证自动扩缩容和负载均衡没有继续摘除实例。

## 常见误判

- 不能仅凭一次 curl 超时认定服务整体不可用，应进行多实例、多入口或多位置探测。
- HTTP 503 可能来自网关无健康后端，也可能是应用主动过载保护。
- 监控指标消失可能是采集链路故障，不一定是服务进程停止。
- 重启后恢复不代表根因解决，仍需检查发布、资源和依赖证据。

## 预防措施

- 健康检查应区分存活和就绪，避免依赖抖动导致全部实例同时摘除。
- 发布采用灰度、自动暂停和快速回滚策略。
- 为关键依赖设置超时、有限重试、熔断和降级。
- 定期演练单实例、依赖故障和入口故障场景。
