# CPU 使用率过高告警处理 Runbook

## 适用场景与检索关键词

- 告警：`HighCPUUsage`、CPU 持续高于 80%、单核打满、系统负载升高。
- 关键词：CPU high、load average、CPU throttling、user time、system time、iowait、死循环、热点进程。
- 适用于 Linux 主机、Docker 容器和普通应用服务；不适用于仅出现瞬时尖峰且业务指标正常的情况。

## 告警定义与影响

建议将 CPU 使用率连续 5 分钟超过 80% 定义为警告，连续 5 分钟超过 90% 定义为严重。高 CPU 可能引起响应变慢、超时、线程调度延迟和实例健康检查失败，但 CPU 高本身只是症状，不是根因。

## 排查前必须确认的信息

1. 服务名、实例名、环境以及告警开始时间和持续时间。
2. 是单个实例、部分实例还是所有实例同时升高。
3. 同期 QPS、错误率、响应时间、内存和发布变更是否异常。
4. CPU 指标是主机口径还是容器口径，容器是否配置 CPU limit。
5. 若用户只说“某服务 CPU 高”，应先查询证据，不能直接认定死循环或流量突增。

## 证据驱动的排查流程

### 1. 确认告警和时间窗口

使用 `get_current_time` 记录诊断时间，使用 `query_prometheus_alerts` 确认是否存在活动告警。若没有活动告警，仍需根据用户提供的时间查询历史日志，但报告中应明确“当前未观察到活动告警”。

### 2. 查询 CPU 趋势

使用 `query_cpu_metrics` 查询目标服务。重点关注平均值、峰值、持续时间、实例间差异和指标是否随请求量同步变化。

### 3. 关联内存和 GC

使用 `query_memory_metrics` 检查内存是否同步上涨。CPU 与内存同时持续上升且日志出现 Full GC 时，优先转入 JVM GC Runbook；CPU 高而内存平稳时，再考虑热点代码、压缩加密或忙等待。

### 4. 查询应用日志

先用 `search_topic_by_service_name` 查找服务对应日志主题，必要时用 `get_topic_info_by_name` 确认主题信息，再用 `search_log` 查询告警前后 15 至 30 分钟。重点检索 repeated error、timeout、Full GC、task started、retry、rate limit 和批处理任务记录。

### 5. 人工补充主机证据

项目当前没有进程级 CPU 工具时，由运维人员执行只读命令并把结果反馈给 Agent：

```bash
uptime
top -b -n 1 -o %CPU | head -n 20
ps -eo pid,ppid,cmd,%cpu,%mem --sort=-%cpu | head -n 20
pidstat -p <pid> 1 5
```

## 现象、证据与结论判断表

| 现象 | 必要证据 | 可支持的判断 |
|---|---|---|
| 所有实例 CPU 与 QPS 同步升高 | CPU 趋势、请求量、无大量异常重试 | 流量增长或容量不足 |
| 单实例单进程长期接近单核 100% | 进程证据、线程栈或火焰图 | 热点代码、死循环或锁自旋 |
| CPU 呈固定周期升高 | 任务启动日志、调度时间一致 | 定时任务重叠或批处理压力 |
| CPU 高且 Full GC 密集 | GC 日志、内存回收后再次快速增长 | GC 压力或内存问题 |
| 容器 CPU 高但主机空闲 | CPU limit、throttling 指标 | 容器配额过低 |
| CPU 高同时 iowait 高 | `iostat` 或 `vmstat` 证据 | 存储等待，不应归为纯计算问题 |

## 常见原因与处理方案

### 流量增长或容量不足

确认请求量和 CPU 同步增长后，可建议限流、缓存热点数据或水平扩容。扩容前需确认下游数据库和 Redis 能承受新增连接，避免把压力转移给依赖服务。

### 热点代码、死循环或异常重试

保存线程栈、性能采样和对应日志，定位热点方法。若问题紧随新版本发布出现，优先评估回滚；不要在没有代码和时间相关证据时直接下结论。

### 定时任务重叠

检查任务开始、结束和锁记录，调整调度时间、增加分布式锁或拆分批次。仅出现周期性 CPU 波峰而业务无损时，不应直接重启服务。

### JVM GC 或容器限流

GC 问题转入 JVM Full GC Runbook；容器限流应评估 CPU request/limit 和节点余量，再调整配额。

## 分级处置与风险控制

- **观察级**：CPU 短时尖峰、错误率和延迟正常，持续观察 10 至 15 分钟。
- **缓解级**：CPU 持续高且延迟恶化，可限流、暂停非核心任务或扩容。
- **高风险级**：只有在单实例失去响应、证据已保留且其他实例能承载流量时，才可申请使用 `restart_service`；该操作必须经过人工审批，并记录原因和回滚方案。
- CPU 超过 95%、错误率持续上升或多个实例失去响应时，立即升级人工值班处理。

## 修复验证

1. CPU 连续 15 分钟回落到基线或低于 70%。
2. P95/P99 延迟和错误率恢复，实例间 CPU 分布合理。
3. 异常重试、Full GC 或周期任务重叠日志停止增长。
4. 扩容或重启后检查下游连接数，确认没有产生新的依赖告警。
5. 观察至少一个原故障周期，确认问题没有再次出现。

## 常见误判

- 不能仅凭一句“CPU 高”就认定存在死循环，必须结合持续时间、实例范围、QPS 和进程证据。
- load average 高不一定等于 CPU 忙，I/O 等待和不可中断任务也会推高负载。
- 单次采样峰值不能证明持续故障，采样窗口和聚合方式必须一致。
- CPU 使用率下降也可能是服务已经崩溃，必须同时验证请求成功率和健康状态。

## 预防措施

- 同时监控 CPU、throttling、QPS、延迟、错误率和实例数。
- 为定时任务增加互斥、超时和最大并发限制。
- 对热点接口定期压测并保留容量基线。
- 发布后设置观察窗口，比较新旧版本的单位请求 CPU 成本。
