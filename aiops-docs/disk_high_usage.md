# 磁盘使用率过高告警处理 Runbook

## 适用场景与检索关键词

- 告警：`HighDiskUsage`、`InodeFull`、磁盘空间超过 80% 或文件系统只读。
- 关键词：disk full、No space left on device、inode exhausted、read-only file system、日志暴涨、Docker overlay2、磁盘 I/O。
- 适用于 Linux 主机、Docker 宿主机、日志盘和数据盘。

## 告警定义与影响

磁盘使用率超过 80% 应预警，超过 90% 或预计短时间内写满应视为严重。空间耗尽会造成日志写入失败、数据库异常、容器启动失败和文件系统损坏；inode 用尽时即使仍有容量也无法创建文件。

## 排查前必须确认的信息

1. 主机、挂载点、文件系统类型、容量、使用率和增长速度。
2. 是空间用尽、inode 用尽、I/O 延迟还是只读挂载。
3. 受影响的是日志盘、数据盘、系统盘还是容器存储目录。
4. 是否存在日志级别变更、备份、core dump、批量导入或镜像构建。
5. 数据和备份的保留策略，哪些文件允许清理以及谁有审批权。

## 证据驱动的排查流程

### 1. 确认告警和相关日志

使用 `get_current_time` 和 `query_prometheus_alerts` 获取活动告警。通过 `search_topic_by_service_name` 找到受影响服务日志主题，再用 `search_log` 查询 No space left、read-only、write failed、I/O error 和日志轮转失败。

### 2. 人工确认文件系统状态

项目当前没有磁盘专用指标工具，以下只读命令需由运维人员执行并返回结果：

```bash
df -hT
df -i
lsblk -f
du -xhd1 /var /opt /data 2>/dev/null | sort -h
find /var/log -xdev -type f -size +500M -printf '%s %p\n' | sort -n
```

### 3. 定位增长来源

比较目录大小和文件修改时间，重点检查应用日志、容器 JSON 日志、临时上传文件、备份、数据库数据、core dump 和已删除但仍被进程占用的文件。

```bash
lsof +L1
docker system df
journalctl --disk-usage
```

### 4. 判断是否伴随 I/O 问题

空间正常但服务仍慢时，应由人工检查 `iostat -xz 1 5` 和 `vmstat 1 5`。高 await、util 或 iowait 属于 I/O 性能问题，不能归结为空间不足。

## 现象、证据与结论判断表

| 现象 | 必要证据 | 可支持的判断 |
|---|---|---|
| `/var/log` 单个文件持续增长 | 文件大小、修改时间、日志级别 | 日志暴涨或轮转失效 |
| 容量有余但无法创建文件 | `df -i` 接近 100% | inode 耗尽 |
| 删除文件后空间未释放 | `lsof +L1` 显示打开的 deleted 文件 | 进程仍持有文件句柄 |
| Docker 目录占用大 | `docker system df` 分类结果 | 镜像、构建缓存、卷或容器日志增长 |
| 文件系统变为只读 | 内核 I/O 错误、挂载状态 | 文件系统或存储设备异常 |
| 磁盘使用率平稳但延迟高 | `iostat` 高 await/util | I/O 饱和，不是空间告警 |

## 常见原因与处理方案

### 日志轮转失效或错误日志风暴

先保留故障时间段样本，再恢复日志轮转、降低异常日志量并修复重复报错。不要直接删除当前日志导致现场证据丢失。

### 临时文件、备份或 core dump 堆积

确认文件归属、保留要求和最近访问时间后再归档或清理。对上传和批处理目录设置生命周期，备份应转移到专用存储。

### Docker 数据增长

区分镜像、卷、构建缓存和容器日志。卷可能包含业务数据，禁止把无差别 prune 作为默认方案。

### inode 或文件系统异常

inode 耗尽应清理大量小文件并修复产生源；文件系统只读或出现 I/O error 时，应停止写入并升级存储人员检查，不应反复强制重新挂载。

## 分级处置与风险控制

- **观察级**：使用率低于 85% 且增长缓慢，建立容量预测和清理计划。
- **缓解级**：归档确认可删除的旧日志或临时文件，恢复日志轮转，必要时扩容。
- **高风险级**：删除、截断、清理 Docker 卷或数据库文件前必须人工审批、备份并确认文件归属。
- 使用率超过 95%、根分区满、文件系统只读或出现 I/O error 时，立即升级人工处理。

## 修复验证

1. 空间和 inode 均回到安全水位，建议低于 75%。
2. 日志、数据库和临时文件能够正常写入。
3. 空间增长速率恢复到基线，日志轮转按预期执行。
4. 已删除文件不再被进程占用，容器和服务健康正常。
5. 若执行扩容，确认文件系统已经识别新增容量。

## 常见误判

- 不能仅凭总磁盘使用率判断哪个服务是根因，必须定位挂载点、目录和文件增长来源。
- 空间未满仍可能因 inode 耗尽而写入失败。
- 删除大文件后 `df` 不下降，可能是进程仍持有文件句柄。
- Docker 占用高不表示所有数据都可清理，volume 可能保存持久化业务数据。
- 磁盘 I/O 高与磁盘容量高是不同故障，需要不同证据。

## 预防措施

- 对空间、inode、增长速率和 I/O 延迟分别设置告警。
- 为应用日志和容器日志配置大小、数量和保留时间上限。
- 对临时文件、备份、core dump 和构建缓存设置生命周期。
- 定期验证日志轮转和磁盘扩容流程。

## 关联告警

- `InodeFull`
- `DiskReadOnly`
- `DiskIOHigh`
- `LogWriteFailed`
- `ServiceUnavailable`
